## Layout & Interaction Patterns

### Navigation & Structure (Phase 2 Extensions)

**Header:** Inherited from Phase 1 (sticky top nav, logo, Dashboard link)

**Job detail sidebar actions (new):**
- Existing: "Edit Job", "Archive Job"
- New: "Start Hunting" button prominently placed at top of actions sidebar
  - State: Primary blue button
  - Loading state: disabled with spinner + "Hunting..." label
  - Disabled state: Visible if hunting already in progress for this job

**Breadcrumbs:** Add link back to job: "Jobs > {Job Title} > Candidates" (for candidate review page)

---

### Company Target List

**Purpose:** Display AI-generated list of target companies for a job. Allow consultant to add, remove, or adjust.

**Structure:**
- Header: "Target Companies" (24px, 600 weight)
- Subtitle: "{count} companies identified" (14px, 400 weight, gray-600)
- List of company cards stacked vertically, gap: md (16px)

**Company Card Layout:**
- Background: white with 1px gray-200 border
- Padding: md (16px)
- Border-radius: 4px
- Content:
  - Company name (16px, 600 weight)
  - Industry (14px, 400 weight, gray-600)
  - Size (14px, 400 weight, gray-600) — e.g. "50-200 employees"
  - LinkedIn URL (as clickable link, blue-600, truncated if long)
- Actions (right side, flex row, gap: sm):
  - View on LinkedIn (icon button, Heroicons external-link)
  - Remove (icon button, Heroicons trash-2, destructive red on hover)

**Interaction:**
- Hover: card shadow increases (same as Phase 1)
- Remove button: Click → inline confirmation "Remove this company?" with "Remove" (red) + "Cancel" (gray)
- Add company (bottom of list): Button "Add Company Manually" → modal with fields: Company Name (required), Industry (optional), Size (optional), LinkedIn URL (optional)

---

### Candidate Cards

**Purpose:** Display AI-discovered candidates per job. Consultant reviews and approves/rejects before outreach.

**Structure:**
- Page header: "Candidates for {Job Title}" (32px, 700 weight)
- Filter tabs (optional, Phase 2.1):
  - "All" | "Approved" | "Rejected" (secondary gray buttons, active = blue accent border-bottom)
- Candidate card grid (1 column mobile, 2 columns tablet, 2-3 columns desktop)
- Gap between cards: md (16px)

**Candidate Card Layout:**
- Background: white with 1px gray-200 border
- Padding: md (16px)
- Border-radius: 4px
- Header row:
  - Candidate name (16px, 600 weight)
  - Status badge (top right) — color per semantic palette above
- Content (stacked):
  - Current title (14px, 400 weight, gray-700)
  - Current company (14px, 400 weight, gray-700)
  - Match score (optional, Phase 2.1): "87% match" (14px, 500 weight, accent blue) — can be added later
- LinkedIn link (14px, blue-600) — clickable icon + "View profile"
- Email (if available): email address displayed as gray-700 (14px), or "Not available" in italic gray-500
- Actions (bottom, flex row, gap: sm, justified):
  - "Edit" button (secondary, smaller)
  - "Approve & Proceed" button (primary blue)
  - "Skip" button (secondary)

**Interaction:**
- Edit click → modal with editable fields: name, title, company, email, LinkedIn URL
- Approve click → card fades out or moves to "Approved" section; if filter active, redirect to Approved tab
- Skip click → confirmation inline "Skip {name}? They won't be contacted." → "Skip" (destructive red) + "Cancel"
- Card hover: shadow increases, cursor changes to pointer if clickable

**Alternate: Card Grid with Inline Actions**
- If cards are full width: Actions stacked vertically (Edit, Approve, Skip) instead of horizontal
- If cards are 2-column: Keep horizontal actions, but smaller button text

### Outreach Drafts View

**Purpose:** Display AI-drafted LinkedIn messages and emails per approved candidate. Consultant reviews, edits, and approves before sending.

**Structure:**
- Page header: "Outreach Drafts for {Job Title}" (32px, 700 weight)
- Filtering (optional):
  - Tabs: "All Drafts" | "Pending Review" | "Approved" (secondary, active = blue accent)
- Draft cards stacked vertically, gap: lg (24px)

**Outreach Draft Card Layout:**
- Background: white with 1px gray-200 border
- Padding: lg (24px)
- Border-radius: 4px
- Header row:
  - Candidate name (18px, 600 weight)
  - Status badge (top right) — "Pending Review" (amber), "Approved" (green), "Sent" (purple)
  - Edit toggle (icon button, Heroicons pencil, secondary)
- Two sections (tabs): "LinkedIn Message" | "Email" (toggle between views)
- Draft content:
  - Read mode: Markdown/plain text display, light gray background (bg-gray-50), padding md, border-radius 4px
  - Edit mode: Textarea with same input styling as Phase 1, height: 200px
- Action buttons (bottom row, gap: md):
  - "Save Edits" (primary blue, appears only in edit mode)
  - "Approve & Queue" (primary green, approval button)
  - "Discard" (secondary gray)

**Interaction:**
- Tab click: Switch between LinkedIn/Email view without losing edits
- Edit toggle: Textarea appears in place of read view; button changes to "Save Edits"
- "Save Edits": Validates content, shows spinner, updates draft, toggles back to read mode
- "Approve & Queue": Marks draft as approved, moves to "Approved" tab if filtering, shows success toast
- "Discard": Confirmation "Discard draft for {name}? You can re-generate it later." → "Discard" (red) + "Cancel"

**Note:** Phase 2 does NOT send outreach automatically. Consultant approves drafts; actual sending (OUTR-03 approval flow) happens in Phase 2.3 via a separate "Send" action on a queue/review page.

---

### Outreach Status Tracking

**Purpose:** Display per-candidate outreach status and history.

**Surface:** Candidate detail view or pipeline card (integrated with candidate cards or separate view)

**Status States (OUTR-04):**
- Pending: Draft created, awaiting approval
- Approved: Consultant approved; queued for sending
- Sent: Message(s) sent successfully
- Replied: Candidate replied to outreach
- Not Interested: Candidate declined or marked as unresponsive
- Error: Sending failed; awaiting retry or manual handling

**Status Card Layout:**
- Row showing: Status badge | Date/time | Last action (e.g. "Sent LinkedIn message")
- If status = Replied: Show conversation snippet or link to view full conversation
- If status = Error: Show error message + "Retry" button

**Interaction:**
- Click status → modal with full history: "Created draft (2026-03-24 14:30)" → "Approved (2026-03-24 15:15)" → "Sent (2026-03-24 16:00)" → "Replied (2026-03-25 09:30)"
- Update status: Inline state button (e.g., "Mark as Not Interested") with dropdown: Replied | Not Interested | Error

## Notes for Executor

1. **Inheritance from Phase 1:** This phase reuses all spacing, typography, colors (base), and component classes from Phase 1. Do not duplicate styles.

2. **New semantic colors (Phase 2):** Add status badge classes for hunting, approved, pending-outreach, sent, replied, not-interested, rejected to `/static/style.css`. Use Tailwind's color utilities (cyan-100, green-100, amber-100, purple-100, emerald-100, gray-100, red-100).

3. **Status enum update:** Add `JobStatus` enum values or create new `CandidateStatus` and `OutreachStatus` enums in Pydantic models:
   ```python
   class CandidateStatus(str, Enum):
       hunting = "hunting"      # AI identified
       approved = "approved"    # Consultant approved
       rejected = "rejected"    # Skipped by consultant

   class OutreachStatus(str, Enum):
       pending_review = "pending_review"     # Draft created
       approved = "approved"                 # Consultant approved
       sent = "sent"                         # Message sent
       replied = "replied"                   # Response received
       not_interested = "not_interested"     # Declined
       error = "error"                       # Send failed
   ```

4. **Candidate Card Grid:** Use Tailwind grid with responsive columns:
   ```html
   <div class="grid grid-cols-1 md:grid-cols-2 lg:grid-cols-2 gap-4">
   ```

5. **Modal Implementation:** Use server-rendered modals with HTMX or simple CSS `display: none/block` toggle. No JavaScript modal library needed.

6. **Edit Toggle:** Use HTMX or Jinja2 conditionals to swap between read and edit modes. Consider `hx-post` to validate and save edits server-side.

7. **Status Badges:** Apply classes dynamically based on status enum value:
   ```html
   <span class="badge-{{ candidate.status }} px-3 py-1 rounded-full text-sm">
       {{ candidate.status.replace('_', ' ').title() }}
   </span>
   ```

8. **Form Validation:** Maintain Phase 1 pattern: server-side validation primary, errors displayed below field in red (`.form-error` class).

9. **Links to LinkedIn:** External links should have icon and text: `<a href="..." target="_blank" rel="noopener noreferrer">`

10. **Accessibility:** All buttons and icons must have descriptive labels. Use `aria-label` for icon-only buttons:
    ```html
    <button aria-label="Remove this company" onclick="...">
        <svg class="w-5 h-5"><!-- trash icon --></svg>
    </button>
    ```

11. **Table option (if needed later):** Outreach status could be displayed in a table view instead of cards. Keep both options in mind but card-based approach is recommended for Phase 2.

12. **Date/time formatting:** Use Python `strftime()` or Jinja2 filters to display in human-readable format: "Mar 24, 2026 at 2:30 PM"

13. **Empty state emphasis:** Ensure empty states are clear and guide user to next action. Use illustration or icon if available (Heroicons illustrations are free).

14. **Confirmation dialogs:** Use HTMX form submission for destructive actions (skip, remove company, discard draft). Always require explicit confirmation via button click, not just alert().
